# Étape 7 – Organisation Agile

## Méthode utilisée

Le projet **Librairie ABC** a été organisé selon une approche Agile inspirée de **Scrum** afin d'améliorer le suivi des tâches, la qualité du développement et l'adaptation progressive aux besoins du projet.

---

## 1. Répartition des rôles

| Rôle | Responsabilité |
|---|---|
| Développeur principal | Développement des fonctionnalités |
| Testeur | Vérification des tests et validation |
| Responsable qualité | Analyse du code avec Pylint, Flake8, Bandit |
| Product Owner | Définition des besoins fonctionnels |

Dans ce projet académique, une seule personne a assuré l’ensemble des rôles.

---

## 2. Backlog des tâches

### Fonctionnalités principales
- Ajouter un livre
- Supprimer un livre
- Rechercher un livre
- Réserver un livre
- Annuler une réservation

### Sécurité et validation
- Validation des données utilisateur
- Gestion des erreurs personnalisées
- Logging des événements sensibles

### Qualité logicielle
- Analyse Pylint
- Analyse Flake8
- Analyse Bandit
- Couverture des tests

### Documentation
- Rapport d’analyse
- Refactoring
- Réécriture des spécifications

---

## 3. Sprint Planning

### Sprint 1 – Développement initial
Objectif :
- Créer la logique métier de la bibliothèque.

Tâches :
- Création des classes
- Gestion des livres
- Recherche et réservation

---

### Sprint 2 – Sécurité et validation
Objectif :
- Renforcer la sécurité du projet.

Tâches :
- Validation des entrées
- Exceptions personnalisées
- Logging
- Analyse Bandit

---

### Sprint 3 – Qualité logicielle
Objectif :
- Améliorer la qualité du code.

Tâches :
- Correction Pylint
- Respect PEP8
- Refactoring
- Couverture des tests

---

### Sprint 4 – Documentation finale
Objectif :
- Produire les livrables finaux.

Tâches :
- Rapport d’analyse
- Documentation technique
- Spécifications mises à jour

---

## 4. Sprint Review

À la fin de chaque sprint, les éléments développés ont été vérifiés :

- Fonctionnalités testées
- Vérification des erreurs
- Validation des résultats
- Contrôle qualité du code

Résultats :
- Flake8 : 0 erreur
- Bandit : 0 vulnérabilité
- Coverage : 96%
- Pylint : problèmes mineurs restants

---

## 5. Rétrospectives

Améliorations identifiées :

### Points positifs
- Projet modulaire
- Bonne couverture de tests
- Sécurité renforcée
- Documentation améliorée

### Points à améliorer
- Réduire les derniers warnings Pylint
